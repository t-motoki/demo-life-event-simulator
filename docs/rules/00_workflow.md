## 全体サイクル

```
要件確認 → 振る舞い仕様 → 設計 → 実装・テスト → レビュー → リリース
```

FPとの合意（計算モデルの前提）が確認できていない状態で設計・実装に進まない。

---

## エージェント一覧

| エージェント | 役割 | 呼ぶタイミング |
|---|---|---|
| `planner` | タスク分割・着手順序・FP確認事項の洗い出し | 実装開始前・方針に迷ったとき |
| `tester` | 振る舞い仕様の定義（テストコードは書かない） | 実装の最初・必ず最初に呼ぶ |
| `architect` | モジュール構成・インターフェース設計 | tester の後 |
| `engineer` | TDDで実装・テスト実行 | architect の後 |

## 実装フロー（TDD）

**原則: 目的が先、手段が後。**

コードの変更は必ず以下の順で進める：

1. **planner**: タスクを整理し `tasks/todo.md` に計画を書く。FP確認が必要な事項を洗い出す
2. **tester**: 振る舞い仕様を記述する（「〇〇のとき□□すると△△になる」）
3. **architect**: tester の仕様を満たす実装構造を設計する
4. **Red**: engineer がテストを先に書く。失敗することを確認する
5. **Green**: テストが通る最小限のコードを書く
6. **Refactor**: 全テストが通る状態を保ちながらリファクタリングする

「テストが通ったから完了」ではない。全テスト通過 + 全FP確認事項の解消 で完了。

## FP確認プロトコル（全エージェント共通）

> **計算モデルの前提はFP実務の判断に基づく。実装者が独断で決めない。**

確認が必要な事項の例：
- インフレ率・物価上昇率の前提値
- 運用利回りの計算方法
- 公的年金の受給額・計算方法
- 住宅ローン控除の適用条件・計算ロジック
- 教育費の標準モデル

### 対応ルール

1. 未確定の数値・計算方法がある場合、実装を止めてユーザーに確認を求める
2. 仮置きする場合は必ず `TODO: FPに確認（仮値）` コメントを付ける
3. 確認事項は `tasks/todo.md` の「FP確認待ち」セクションに記録する

---

## 共通ルール（全エージェント共通）

### コードの品質

- **シンプル第一**: 影響するコードを最小限にする。過剰な抽象化をしない
- **1モジュール = 1責任**: 入力・計算・出力を明確に分離する
- **計算ロジックは純粋関数**: 副作用なしでテストしやすくする
- **コメントは「なぜ」を書く**: 「何を」はコードを見ればわかる
- **エラーメッセージは日本語**: FPが読める言葉で出力する

### ドキュメント

- 計算モデルに関わる設計判断は `docs/` に記録する
- 要件が変わった場合は `docs/spec/01_requirements.md` を更新し、最終更新日を変える
- 学びは `tasks/lessons.md` に記録する

### Git

- コミットメッセージは日本語
- 意味の単位で分けてコミットする（ドメイン変更・インフラ変更・設定変更など）
- `src/` 以下の変更には必ずテストを通してからコミットする

## ディレクトリ構成

```
src/
  domain/        ← 計算ロジック・エンティティ（外部依存なし）
  input/         ← 入力データの読み取り・バリデーション
  output/        ← Excel 出力
  main.py        ← エントリーポイント
tests/
  domain/        ← 計算ロジックのテスト（最重要）
  input/
  output/
docs/
  rules/         ← ワークフロー・共通ルール（このファイル）
  spec/          ← FPと合意した仕様・要件
  guides/        ← 非エンジニア向けの操作手引き
tasks/
  todo.md        ← 現在のタスク・FP確認待ち事項
  lessons.md     ← 過去の学びと教訓
```
