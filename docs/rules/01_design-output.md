# 設計図の出力ルール

作成日: 2026-04-07

---

## 原則

設計図・構造図・モデル図はすべて **Mermaid 形式** で出力する。

---

## 理由

- テキストで説明するより構造が一目で伝わる
- Mermaid はコードとして管理できる（差分が取れる）
- `docs/` に `.mmd` ファイルとして保存することで設計判断が追跡可能になる

---

## 図の種類と使い分け

| 図の目的 | 使う図種 | Mermaid キーワード | 言語 |
| -------- | -------- | ------------------ | ---- |
| システム全体構造（レイヤー・依存関係） | フローチャート | `graph TD` | 日本語 or 英語 |
| ドメインモデル（業務概念・関係） | フローチャート | `graph LR` | **日本語（ユビキタス言語）** |
| クラス図（コード構造） | クラス図 | `classDiagram` | **英語（コードと一致）** |
| ユースケース（誰が何をするか） | フローチャート | `graph LR` | 日本語 |
| 処理の流れ・分岐 | シーケンス図 | `sequenceDiagram` | 日本語 or 英語 |

**ドメインモデル図とクラス図は別物:**

- ドメインモデル図: FPとエンジニアが共通認識を持つための業務概念図。ユビキタス言語（日本語）で記述する
- クラス図: ソースコードの構造を表す技術図。英語でコードと一致させる

ドメインモデル図にはソースコードへの参照（`%% → models.py: Scenario` 等）をコメントで添えることを推奨する。

---

## 保存先

```
docs/diagrams/
  system.mmd      ← システム全体構造
  domain.mmd      ← ドメインモデル
  usecase.mmd     ← ユースケース（必要に応じて）
```

---

## 出力時のルール

1. **ファイルとして出力する**: チャット内に貼り付けるだけでなく、必ず `docs/diagrams/` に `.mmd` ファイルとして保存する
2. **コメントを入れる**: ビジネスルールに関わるノードには `note` または `%%` コメントで補足を入れる
3. **FP確認済みルールは図に含める**: 住宅ローン控除の年数など、確定した前提値をラベルに明記する
